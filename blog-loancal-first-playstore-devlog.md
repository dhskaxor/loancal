# 첫 구글 플레이 등록 후기 🚀 대출계산기 앱 만들면서 배운 것들

드디어 첫 앱을 구글 플레이 콘솔에 올렸습니다.  
솔직히 말하면, 기능 개발보다 배포 단계에서 더 많이 배웠어요 😅

이번 글은 제가 대출계산기 앱을 만들면서 겪은 시행착오를 그대로 정리한 개발기입니다.  
"앱이 돌아간다"와 "스토어에 올릴 수 있다"는 정말 다르더라고요.

## 시작: 왜 대출계산기였나 💰

주변에서 대출 이야기할 때 늘 비슷한 질문이 나왔어요.

- 매달 얼마 내야 하지?
- 총 이자는 얼마나 붙지?
- 원리금균등, 원금균등, 만기일시 중 뭐가 나한테 맞지?

그래서 첫 버전은 욕심 줄이고 MVP로 갔습니다.

- 3가지 상환 방식 계산
- 계산값 저장/불러오기/비교
- 월별 상환 스케줄 표

딱 "실제로 쓸 수 있는 계산기"를 목표로 잡았어요.

## 구현 방식: 웹 + 앱 조합으로 빠르게 ⚙️

앱 화면을 처음부터 전부 네이티브로 만들기보다,  
웹(React)에서 계산 로직/화면을 먼저 완성하고 Expo WebView로 앱에 붙였습니다.

- 웹: 계산 UI, 결과 카드, 비교표, 스케줄
- 앱: WebView + 광고 + 스토어 배포

이 방식의 장점은 명확했어요.  
웹에서 수정하면 앱에도 바로 반영되는 흐름이라, 초반 반복 개발 속도가 훨씬 빨랐습니다.

## 광고는 "기술"보다 "운영" 문제였다 📣

처음엔 AdSense를 생각했는데, 승인 대기 기간이 변수라서  
우선 카카오 애드핏으로 운영 가능한 상태를 먼저 만들었습니다.

그리고 나중에 승인되면 바꿀 수 있도록 분리해뒀어요.

- 하단 배너: 상시 노출
- 전면 광고: 계산 이벤트 누적(4, 8, 12...) 기준 노출

결국 중요한 건 "내가 당장 운영 가능한 구조"를 먼저 만드는 거였습니다.

## 실제로 막혔던 포인트 2개 🧩

### 1) 스케줄 표 오른쪽이 잘리는 문제

모바일에서 상환 스케줄 표를 보면, 오른쪽 열(잔금)이 잘려 보이는 케이스가 있었어요.  
처음엔 숫자 데이터 문제인 줄 알았는데, 원인은 UI 폭/줄바꿈 처리였습니다.

해결은 이렇게 했습니다.

- 스케줄 영역 가로 스크롤 허용
- 숫자 셀 줄바꿈 방지
- 모바일 기준 셀 간격/폰트 크기 재조정

작은 UI 깨짐이지만, 사용자 입장에서는 "계산이 틀린가?"로 느껴질 수 있어서 꼭 잡아야 했습니다.

### 2) Play Console 버전 충돌

첫 등록 때 제일 많이 본 문구가 버전 충돌이었습니다.  
핵심은 `versionName`이 아니라 `versionCode`였어요.

- `versionCode`는 업로드할 때마다 반드시 증가
- `versionName`은 사용자에게 보이는 버전
- 매번 AAB 재생성 후 업로드 확인

당연한 규칙인데, 실제 배포하면서 몸으로 익혔습니다 😂

상환 스케줄 화면

### 3) 비교 계산 화면

비교 계산 화면
*여러 조건을 한 번에 비교해서 의사결정 시간을 줄이는 화면입니다.*

## 첫 등록 체크리스트 ✅

- 릴리스 서명/키 설정 확인
- `versionCode` 증가 확인
- AAB 생성 후 콘솔 업로드 테스트
- 스토어 그래픽/스크린샷 준비
- 광고/개인정보 관련 문구 점검

## 이번 출시에서 배운 점 3가지

1. **개발 완료와 출시 준비 완료는 다르다.**
  기능 구현보다 배포 체크리스트가 더 중요할 때가 많았습니다.
2. **광고는 코드만 넣는다고 끝이 아니다.**
  승인 시간, 노출 위치, 사용자 이탈까지 같이 봐야 운영이 됩니다.
3. **모바일 UI의 작은 깨짐은 신뢰 문제로 이어진다.**
  특히 숫자 표 같은 화면은 정렬/가독성이 정말 중요했습니다.
